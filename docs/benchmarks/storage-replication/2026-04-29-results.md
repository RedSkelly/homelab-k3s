## Bottleneck Analysis

| Layer | Throughput | Notes |
|---|:-:|---|
| VLAN 20 (raw network) | ~9.07 Gbps (~1,134 MB/s) | Near 10GbE line rate |
| SATA SSD (theoretical) | ~500 MB/s | Typical SATA III ceiling |
| Longhorn seq write | 94.3 MB/s | 5.3x below SATA SSD max |
| Longhorn seq read | 132.2 MB/s | 3.8x below SATA SSD max |

**The bottleneck is Longhorn overhead, not the network or disk.** The VLAN 20 fabric delivers >9 Gbps — far more than needed. Even a single SATA SSD should sustain ~500 MB/s sequential. Longhorn's sequential write at 94 MB/s represents significant overhead from:

1. **3-replica synchronous replication** — each write must commit to 3 nodes before ack
2. **Longhorn's iSCSI target layer** — adds latency per I/O
3. **Default iodepth=1** on the sequential tests — Longhorn's pipeline isn't saturated (a higher `iodepth` or `numjobs` would likely improve throughput)

## Anomalies Flagged

- **High TCP retransmits on all VLAN 20 pairs** (55-68K over 10s). Despite throughput being good, this volume of retransmits suggests suboptimal TCP tuning — likely default socket buffers are too small for 10GbE, or the MTU is 1500 instead of jumbo frames (9000). Enabling jumbo frames on VLAN 20 and tuning `net.core.rmem_max` / `net.core.wmem_max` would likely reduce retransmits and could improve Longhorn replication latency.

## Follow-up Ideas
 
- Re-run sequential tests with `--iodepth=16` or `--numjobs=4` to isolate how much of the gap is pipeline starvation vs Longhorn overhead.
- Enable jumbo frames (MTU 9000) on VLAN 20 and re-benchmark to measure retransmit and latency impact.
- Both would be good candidates for additional dated results files in this directory.
